# TESTING_POLICY.md — سياسة الاختبار المستمر (ملزمة)

## المبدأ الأساسي

**لا يُعتبر أي Sprint "مكتملًا" حتى يجتاز اختبار قبول (Acceptance Test)
موثّقًا — بغض النظر عن مدى اكتمال الكود.** هذا ينطبق بأثر رجعي على كل ما
بنيناه، وبشكل إلزامي على كل ما يُبنى بعد هذا الملف.

## لماذا هذا ملزم لا اختياري

اكتشفنا عمليًا (اختبار المستخدم اليدوي لصفحة التسجيل) أن الكود "الصحيح
نظريًا" قد يُخفي فجوات حقيقية (مثل: التنسيق المفقود، لوحة معهد تعرض واجهة
طالب). الاختبار اليدوي بعد كل مرحلة هو ما كشف هذه الفجوات — وليس مراجعة
الكود وحدها.

## بوابة القبول لكل Sprint (Definition of Done الموسّع)

كل Sprint يُغلق فقط بعد اجتياز الخطوات الثلاث التالية بالترتيب:

### 1. اختبار وظيفي يدوي (Functional Walkthrough)
المالك (أو مختبِر) يمر يدويًا على كل ميزة جديدة بالسيناريو الطبيعي
والسيناريو الفاشل عمدًا (بيانات خاطئة، إلغاء منتصف عملية، رابط غير
موجود). يُوثَّق بلقطات شاشة أو وصف نصي لكل حالة + النتيجة (نجاح/فشل).

### 2. فحص الوثائق (Docs Sync Check)
هل تغيّر أي شيء معماري/أمني/في قاعدة البيانات؟ إن نعم، هل تحدّثت
الوثيقة المقابلة (ARCHITECTURE.md / SECURITY.md / DOMAIN_CONTRACTS.md /
ROADMAP.md) في نفس الدفعة؟ الوثائق المتأخرة عن الكود = دين تقني، وتُمنع.

### 3. فحص بناء نظيف (Clean Build Check)
`npm run build` و`npm run lint` و`npm run test` تمر بلا أخطاء على نسخة
حديثة مسحوبة من GitHub (لا نسخة محلية قد تحوي إصلاحات غير مرفوعة).

**لا ننتقل للـ Sprint التالي إلا بعد اجتياز الثلاثة.**

## خطة اختبار Beta المصغّر (بعد اكتمال Sprint 3)

هدفها: كشف مشاكل الأداء والاستخدام الحقيقي قبل أي استثمار في سعة إضافية
(Redis، ترقية Supabase) — نقيس أولًا، ثم نطوّر بناءً على بيانات حقيقية لا
تخمين، بنفس فلسفة تأجيل الاشتراكات سابقًا.

### النطاق
- 5–10 مستخدمين حقيقيين (أصدقاء/عائلة)، خليط من الفئات (طالب، باحث عن
  عمل، موظف على الأقل)
- مدة أسبوع واحد
- بيئة: نفس Supabase الحالي (الخطة المجانية) — لا ترقية مسبقة

### معايير النجاح (Success Criteria) — محدّدة رقميًا لا انطباعيًا
| المقياس | الحد المقبول |
|---|---|
| نجاح التسجيل الكامل (تسجيل→onboarding→dashboard) | 100% من المختبرين بلا مساعدة يدوية مني |
| زمن استجابة Dashboard | أقل من 3 ثوانٍ على اتصال عادي |
| أخطاء غير معالجة (رسالة تقنية خام تظهر للمستخدم) | صفر حالات |
| نجاح إكمال درس واحد + ظهور النقاط فعليًا | 100% |
| استقرار محادثة الوكيل (بلا انقطاع 502/timeout) | ≥ 90% من المحاولات |

### ماذا نفعل بالنتائج
- كل مشكلة تُسجَّل كسطر في TECH_DEBT.md بدرجة خطورة
- أي فشل في معيار النجاح = يُحل قبل دعوة دفعة Beta أوسع، وليس قبل أي
  ميزة جديدة أخرى
- بعد اجتياز المعايير: ننتقل بثقة لتوسيع القدرة (Redis/ترقية Supabase)
  بأرقام حقيقية تبرر القرار، لا تخمينًا
