# CLAUDE.md — تعليمات جلسات Claude Code لمشروع WOW

> اقرأ هذا الملف أولًا في كل جلسة. هو نقطة الدخول لفهم المشروع كاملًا.

## ما هذا المشروع

**WOW (World of Work)** — منصة تعليمية تأهيلية توظيفية ثنائية اللغة (AR/EN):
تعليم → توظيف → ترقية، مع وكيل ذكاء اصطناعي شخصي اسمه **Nova**، ونظام
Career DNA (توأم رقمي مهني)، ومحرك أدلة للمهارات، وعقود تشغيل خارجي
بضمان المنصة عبر شريك مرخّص.

المالك: MonzerHussan — المستودع المستهدف: `https://github.com/MonzerHussan/WoW`

## الحالة الحالية (لحظة كتابة هذا الملف)

**منجز ككود، غير مشغَّل بعد على بيئة المالك:**
- تطبيق Next.js 14 كامل: auth (signup/login/logout)، middleware حماية،
  onboarding، dashboard، Nova chat، نظام نقاط — بنية features/ + shared/
- قاعدة البيانات: 6 ملفات SQL يجب تشغيلها **بالترتيب الصارم**:
  `supabase/schema.sql` ثم `supabase/migrations/002..006`

**لم يُنفَّذ بعد (مهامك المحتملة الأولى):**
1. `npm install && npm run build` — تحقق أن البناء يمر (لم يُجرَّب فعليًا
   لأن بيئة التطوير السابقة كانت بلا إنترنت — أصلح أي خطأ types/imports
   صغير قد يظهر)
2. تنفيذ الـ SQL على مشروع Supabase الخاص بالمالك (بالترتيب أعلاه)
3. تعبئة `.env.local` من `.env.local.example`
4. `npm run dev` والتحقق من: signup → onboarding → dashboard → Nova
5. الرفع إلى GitHub (`MonzerHussan/WoW`)

## قواعد غير قابلة للكسر (ملخص — التفصيل في CODING_GUIDELINES.md)

1. **اتجاه الاستيراد**: `app → features → shared` فقط. الميزات لا تستورد
   من ميزات شقيقة (التركيب المشترك يحدث في صفحات app/).
2. **لا نصوص AR/EN مكتوبة داخل المكونات** — كل النصوص في
   `shared/i18n/dictionary.ts` عبر `t()` أو `useLang()`.
3. **كل مدخل API يُتحقق منه بـ zod** من `shared/schemas/` قبل لمس
   Supabase أو OpenAI.
4. **النقاط لا تُقبل من العميل أبدًا** — فقط `reason` يُبحث في
   `shared/constants/points.ts` (كانت ثغرة حرجة وأُصلحت — لا تُعِدها).
5. **الأسرار خادمية فقط** — لا `NEXT_PUBLIC_` على أي مفتاح سري.
6. **تغييرات قاعدة البيانات = migration جديد مرقّم** في
   `supabase/migrations/` — لا تعدّل ملفات migrations القديمة أبدًا.
7. **ميثاق الشفافية T1–T9** (في DOMAIN_CONTRACTS.md) ملزم: كل درجة لها
   تفسير إلزامي، بيانات الشخصية لا تُشارك مع المنظمات أبدًا، الموافقة
   لكل منظمة على حدة، الدرجات ترشّح ولا ترفض آليًا.
8. **الوثائق تتحدث مع الكود**: أي تغيير معماري/أمني/أدوار يحدّث الوثيقة
   المقابلة في نفس الـ commit.

## السبرنت التالي المتفق عليه: Sprint 3 — LMS + الوكيل الشخصي + الملف الشخصي + تغذية DNA

المجال الكامل جاهز (migrations 004-006-007-007b-008، نُفِّذت جميعها على
Supabase). المطلوب بناء الواجهات والخدمات، **بهذا الترتيب بالضبط**
(كل مرحلة أساس لما بعدها):

### 1. `features/lms/` (الأساس)
كتالوج الدورات (المنشورة فقط للزوار)، صفحة الدورة، مشغّل الدروس، أخذ
الاختبارات. **كل مستوى PMP الآن hybrid إلزاميًا** (008): تصحيح آلي فوري
+ اعتماد بشري نهائي من مقيّم، بمهلة مراجعة 48 ساعة (`review_deadline`).

### 2. `features/agent/` (الوكيل الشخصي — ليس Nova بعد الآن)
- `user_agent_profiles` (007b): صف واحد لكل مستخدم، ذاكرة موحّدة عبر كل
  قدراته — **لا وكيل منفصل لكل دور**.
- أول محادثة: المستخدم يختار اسم وكيله بنفسه (`chosen_name`). **لا اسم
  مفروض من النظام، ولا تعرض "Nova" كاسم افتراضي ظاهر للمستخدم** — استبدل
  أي إشارة متبقية له في الواجهة بحقل اختيار الاسم عند أول تفاعل.
- الوكيل "يفعل لا يتحدث فقط": يقرأ DNA كامل (مهارات، تسجيلات، درجات)
  ويكتب توصيات فعلية في `career_recommendations`، لا ردودًا عامة فقط.
- أسلوبه يتكيّف حسب القدرة النشطة (طالب/موظف/مستقل...) — هويته لا تتغيّر.
- **الوكيل يبقى مجانيًا الآن** (محميًا فقط بـ rate limit التقني الحالي).
  **لا تربط `spend_coins()` بـ`/api/nova` بعد** — يُفعَّل لاحقًا في سبرنت
  الاشتراكات مع بوابة دفع فعلية.

### 3. `features/profile/` (صفحة `/profile` — إلزامية هذا السبرنت)
تجمع بصريًا: محاور Career DNA، المهارات وأدلتها (`skill_evidence`)،
الشهادات، Trust Score وEmployability Score بتفسيرهما الكامل
(`{factors:[{name,weight,value,tip}]}`)، القدرات النشطة
(`user_capabilities`) مع زر "فعّل قدرة جديدة" (**لا يحذف أي قدرة قائمة
أبدًا**)، والوكيل الشخصي مع آخر توصياته ونتائجها.

### 4. تفعيل عقد LMS→DNA (البند §5 في DOMAIN_CONTRACTS.md)
- إكمال درس → `lesson_progress` + نقاط `LESSON_COMPLETE` (تحقق خادمي
  من الإكمال الفعلي قبل منح النقاط — TECH_DEBT #10)
- اجتياز اختبار (**بعد اعتماد المقيّم**، لأن كل الاختبارات hybrid الآن) →
  `entity_skills(source='assessment')` + `skill_evidence('quiz_attempt')`
  تلقائيًا
- إصدار شهادة → `certificates` + إعادة حساب Employability
- كل إعادة حساب درجة = **صف جديد** في `career_scores` (سلسلة زمنية، لا
  تحديث في المكان) مع `explanation` إلزامي

### بنية المحتوى (008) — غير إلزامية الآن
`lessons.translations` (jsonb، AR/EN)، `content_review_votes` لحوكمة
النشر (owner + peer_assessor + nova_check gate) — ليست ضرورية لبناء
مشغّل الدروس الأساسي، لكن اربط بها إن بُنيت أي واجهة لإدارة المحتوى.

### قاعدة تسمية صارمة: نقاط ≠ كوينز
`profiles.points` ("نقاط"/gamification) و`wallets` ("كوينز"، 007b)
**نظامان منفصلان تمامًا** في كل نص/متغير/مفتاح i18n. لا تُترجم "Coins"
إلى "نقاط" أبدًا في أي مكان، ولا العكس.

### مهمتان محمولتان (منفصلتان عن الترتيب أعلاه)
(أ) middleware يعامل `profiles.status != 'active'` كغير مسجّل؛
(ب) نقل rate limiter من الذاكرة لمخزن مشترك (Upstash) قبل توسيع الوكيل.
